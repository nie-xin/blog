同步I/O与异步I/O

NODE核心概念

同步I/O与异步I/O

Node.js自诞生之日起就以起高并发得名，其中的核心便是异步I/O了。那么到底什么是异步I/O？和传统的同步I/O比又有什么优点？让我们从简单的概念讲起吧。

同步I/O

在操作系统中我们都学过，线程执行在遇到I/O操作的时候会被阻塞。然后CPU转而去处理该I/O操作，得出结果后再返回该线程继续执行下去。这就是传统意义上得同步I/O，也是在编程中非常直观的方式。假设我们有一个名为“test.txt”的文件，下面让我们写一个读取文件的小例子：
// fs是node的file system
var fs = require(‘fs’);
// sync指明同步I/O方式
var data = fs.readFileSync(‘test.txt’, ‘utf-8’);
console.log(data);
console.log(‘end of program.’);

可以预见，该程序先读取test文件的内容并显示在终端上，然后打印一行end of program作为程序的结束。一般程序都是这么线性执行的，理解起来也非常容易。这么执行有什么问题？

问题

这样的线程执行方式在一般情景下是完全没有问题的，但我们将场景切换到互联网上来看。传统服务器的处理方式是为每个请求开启一个线程，在遇到I/O请求的时候就按上面说的同步方式来坐阻塞处理。但每个CPU能承受的线程数是有限制的，于是达到限制的时候就必须添加新的CPU。何况开启线程是非常消耗资源的，访问量教大的互联网应用，服务器扩展的速度可能会远远超过你的想象。比如前两天听的PODCAST中，一个有几十万用户的ROR应该就已经消耗了二十几台服务器了。

策略

那么有其他的解决方法么？从上面的模型上来开，瓶颈主要是在阻塞上。于是异步I/O的概念就此诞生了。与同步想法，异步方式在遇到I/O操作时不阻塞线程，而是将I/O请求发给后台执行，然后线程继续执行下去。直到I/O操作完成后再通知线程来处理结果。于是线程的利用率就很高了，单个线程就能处理大量的请求。下面来看看异步方式I/O到底是什么样的：

var fs = require('fs');
// node默认是异步处理凡事
fs.readFile('test.txt', 'utf-8', function(err, data) {    
// 读取出错则打印出错信息，否则打印文件内容
if (err) {
    console.log(err);
} else {
    console.log(data);
}
});
console.log('end of program.');

这个程序的执行结果又是什么？这次不再是先打印文件内容了，而是先打印了end of program，然后才是文件内容。就像上面提到的，但程序执行到读取文件这个I/O操作的时候，不会再阻塞并等待了。去读操作被交给后台去处理，而线程继续执行下去，也就是执行后面的end of program这条指令。文件读取完成后再打印文件的内容。

缺点

看起来好像异步方式是个非常完美的解决方案。别急着下结论，任何事情都有好的一面和坏的一面，没有任何方案是绝对完美的。异步方式的缺点，就在于其非线性的执行方式带给编写程序的困难。从上面的例子可以看到，异步程序的执行结果与传统的程序有很大区别。这就说明了编写异步程序比较困难，需要改变传统的程序设计思想。别低估了这个困难，传统的力量是远远超出人的想象的！

结语

传统的编程方式在互联网不断扩张之后遇到了不少挑战。并发处理，可扩展性和降低成本也越来越受到重视（可以参考LINKIN招聘数据）。异步I/O无疑是一种（但非唯一）有效的解决方式。

稍后我们在一起看看采用异步I/O方式的NODE到底是如何利用异步模型来处理大规模并发的。到底是什么神奇的力量能让服务器数量从三十台降到三台？

到底怎么精通技能？

到底要怎么样才能精通技能？

越来越怀疑自己，到底我都学会了些什么？许多东西看的时候明白了，跟着例子也会做了，但一回头就忘记了。写程序是这样，学理论也是这样。所以每次考试都要考大量重复来过关，之后就几乎全部忘记了。但写程序，大脑能重复多少次这样的过程？程序也不是考试，没有过关这个说法。现在这个样子，到底什么时候才能做一个熟练的程序员？

我知道，我写的代码还太少；我也知道，我是个爱偷懒的人；我更知道，我的年纪已经不小。但我不想从此放弃，我不想从此挣扎在芸芸众生中。我并不是非常有野心，但至少，我想做一个熟练的程序员而已。

为什么PHP语法看过好几次，还是不能全部记清楚。为什么SYMFONY的结构看了又忘，忘了又看？到底什么时候我才能投入实际生产的开发，自己的代码真正贡献出能量？

我每一天都要问这样的问题，如何才能找出答案？看项目源代码？别天真了，就现在的水平，根本看不懂稍大点项目的代码。或许我该从micro的项目开始？但时间怎么办？后面一忙起来，我真能坚持每天都PUSH代码么？

太多的问题，却永远没有答案。人生是苦，何处是解脱？

Life is life

再过两个月我就正式三十岁了，人生的三分之一已经悄然逝去。我的人生在这两年里已经悄然改变，曾经幻想的生活也已经幻灭。往往回顾那些年，总觉得有些值得回忆的东西，不舍得他们就这样消失在时光里。于是稍微整理一下回忆，整理成我的小故事。

缘起

时光倒流到96年，还满脸稚气的自己参阅者一本厚度近似辞海的打印文档——《DOS参考手册》。这是当时我唯一能接触到关于怎么使用那台神奇的机器，我的第一台468电脑的参考资料。当时自己的知识水平几乎不可能读懂翻译的异常晦涩的手册的，于是只能按例子一个个输入看会有什么结果。到也渐渐熟悉了那几个常用操作，什么dir啦cd啦，也能分辨出能执行的程序文件了。只不过当时的软件少得可怜，除了一个UCDOS能用来联系打字排版之外，就剩下几个盗版的台湾教育软件能玩玩了。就算是如此，那种充满了互动的软件过程还是深深吸引了我。和家里的游戏机的卡带相比，至少我能看到软件里的各种文件了！软件不再是封装在电子版里的黑盒子，而是可以真真看到名字类型的文件了！这对我无疑是一个巨大的触动，虽然我还是不懂那些文件是怎么弄出来并神奇的运行成有图像声音的程序的。但那种贴近的感觉很强烈，也第一次明白了程序原来是有人写出来的，而不是制造成的电路板。

当时还是有编程语言的，就是老式的basic语言。以当时我的智商和耐性，我根本没可能学会这个东西。但有一次我还是依葫芦画瓢地把参考书上得范例输入了进去，居然最后也成功了……可惜这也是唯一一次的成功。我还清楚的记得那是一个简单的小动画，是一片叶子有水滴划过。今天看来是在弱智，但当时的我可是高兴了小半天呢。毕竟是自己第一次写的程序，尽管我也不知道自己写了些什么东西上去，但至少能用了嘛。

晚些时候还挑战过一些比较复杂的程序，但自己缺乏耐心和细致，加上编程环境没有错误提示，也就都不了了之了。当时想过，那些写出程序的人真了不起，一个程序里少说也有上百个文件呢。可自己并未想过以后会做程序员，年少时谁会羡慕整天埋头玩电脑的人，狂霸酷拽吊炸天才是骚年们追求的风格。于是写程序这种事就搁置了下来，很久没有出现在我的生活中了。

追游戏的少年

自小我就是个喜欢游戏的孩子。还记得第一次接触FC时那种肾上腺激素激增的感觉，那种兴奋异常并忘记所有烦恼的快乐。我痴迷于魂斗罗、坦克大战、冒险岛、马里奥等一系列经典游戏中，那些游戏画面组成了我童年不可磨灭的记忆。

当时觉得游戏是一个高科技的东西，画面漂亮，而且一玩就上瘾。
